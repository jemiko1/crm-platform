**Purpose:** Full-system snapshot of the working state **before** FreePBX REST API
integration work begins. Allows one-prompt restore to this exact state.

## One-prompt restore

To roll back, tell Claude Code:

> **Restore telephony baseline 2026-04-24**

The assistant will run `scripts/restore-telephony-baseline.sh 2026-04-24` after
verifying all backup files are present and their SHA256 hashes match. Any mismatch
aborts the restore before touching production.

## What's NOT backed up (and why)

- **`/var/spool/asterisk/monitor/`** (call recordings) — too large, already backed up via recording SCP fetch to CRM's cache directory. Not relevant to config rollback.
- **`/var/log/asterisk/`** — logs regenerate; rolling them back would hide diagnostic data rather than help.
- **Git working tree on VM** (`C:\crm\`) — recreated on redeploy via `git fetch + reset --hard`. The tag handles this.
- **Softphone releases** — we're not altering them in this work stream; if needed, separate rollback via GitHub releases.
- **Core-sync-bridge state** — stateless service; its in-memory checkpoint rebuilds from `SyncCheckpoint` rows which ARE in the Postgres dump.

## What a restore does NOT undo

- **Calls that happened after baseline** (Oct 24 00:35 onward) — their `CallSession`/`CallLeg` rows will be removed by the Postgres restore. If a real customer called in between baseline and restore, we lose that record. Mitigation: during development work, take incremental backups; if a real incident happens, we can selectively preserve the newer rows.
- **FreePBX GUI settings changed via the web interface after baseline** — also wiped by MariaDB restore.
- **Softphone client cache / session state** — operators may need to re-log-in.
- **Alamex trunk provider account state** — external, unaffected by CRM/PBX rollback.

## Restore procedure (what `restore-telephony-baseline.sh` does)

1. **Precheck** — verify all SHA256 hashes match; abort if any mismatch.
2. **Announce** — echo every action it will take; ask operator to confirm at the terminal if `--interactive` is passed (default: non-interactive, all steps run).
3. **Stop services** on VM — `pm2 stop crm-backend crm-frontend`.
4. **Restore Postgres** — `pg_restore --clean --if-exists --no-owner -U postgres -d crm crm-db.dump`.
5. **Restore FreePBX MariaDB** — SSH to asterisk, `mysql -u root asterisk < freepbx-asterisk-db.sql`.
6. **Restore /etc/asterisk** — extract tarball in-place, preserve owners.
7. **Restore ASTDB** — stop Asterisk momentarily, copy file, start.
8. **fwconsole reload** — make Asterisk re-read config.
9. **Restore backend .env** — copy `backend.env.bak` to `C:\crm\backend\crm-backend\.env`.
10. **Git reset** — on VM's `C:\crm\`, `git fetch origin && git reset --hard baseline-pre-pbx-integration-2026-04-24`.
11. **Restart services** — `pm2 restart crm-backend crm-frontend`, wait 15 s, `pm2 logs` tail for errors.
12. **Health checks** — `curl /health` on backend, `pjsip show endpoints` on PBX, AMI bridge `/health` → confirm `ami.connected: true`.
13. **Report** — summary of what was restored, timestamps, any issues.

If any step fails, the script halts, reports what succeeded and what failed, and
leaves the system in whatever state the failure produced — it does not attempt
automatic rollback of a partial restore. (If a restore itself fails, we have
bigger problems than this document can solve — call the engineer.)
